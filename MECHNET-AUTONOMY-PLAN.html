2 · Target behavior — "ask or know", made mechanical

Autonomy is a policy, not a vibe. The door routes on four inputs it can own end-to-end: payload size (it packs the files, it knows), task class history (capacity/findings projections), rung state (occupancy, warmth, credit runway), and a caller-declared quality tier. The contract:

conditiondoor behavior
payload fits a rung + task class has known-good history + quota/occupancy OK KNOW — route silently, return result with routed_by proof
chosen rung fails (cold, 5xx, timeout) KNOW — climb one rung, once; routed_by:"escalation:oxen→flash"
novel task class, or quality tier "best", or trial-credit reserve breached ASK — return a recommendation + reason instead of burning; caller (agent or Derek) decides
architecture, multi-file judgment, whole-conversation context ("Claude keeps" list) NEVER ROUTED — unchanged hard boundary from the strategy doc §3

gcp-gemini-pro stays out of auto-routing's reach (recommend: pin/ASK only — it's the premium reach and the trial credits are finite). Flash remains the opportunistic ceiling; sunk rungs the floor.

Queue-and-forget, defined: a session submits N self-contained briefs in one motion, gets a manifest back, and collects file paths — never result text — when it (or a later session) checks in. Progress is visible on the dashboard, not by polling from frontier context.

5 · Track OXEN — dual-B70 Linux re-baseline (parallel lane, runs on AM4)

Why it's in this plan: the router can only trust am4-oxen with big work if its capacity is data (E6). The campaign's output is literally A1/A3's routing table for that rung — plus the unmeasured Linux dividend (P2P between cards, driver gains) and a safe answer to the 32 GB OOM fear. Methodology drafted by flash through the door (receipt §7), edited here. Respects the standing directive: drive the imagegen pipeline's managed units, never crudely kill oxen; benchmark windows are occupancy-gated.

5.1 Campaign questions

  1. Q1 — What does Linux P2P net us? Windows had no cross-card path; Linux/Level-Zero may. Measure card↔card bandwidth/latency, then the end-to-end effect: single card vs two-independent vs tensor-split vs layer-split, same model+quant.
  2. Q2 — What did the drivers net us? Prefill tok/s, decode tok/s, TTFT, load time at matched model+quant+context vs the Windows-era baselines (denning / b70tools corpus).
  3. Q3 — What is the safe dual-card envelope under 32 GB host RAM? mmap vs no-mmap, quant level, staged load order, swap headroom → a preflight number, not a fear.

5.2 Work packages
