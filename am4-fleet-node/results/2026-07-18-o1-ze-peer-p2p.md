# O1 — ze_peer P2P microbenchmark, dual Arc Pro B70 (AM4)

2026-07-18 · receipt `br-20260718-060412-4f780d5d` · raw logs in `raw-2026-07-18-o1-ze-peer/`

## 1. Context

O1 of `MECHNET-AUTONOMY-PLAN.html` (oxen re-baseline). O0 (2026-07-17) was the inventory
sweep: it found the `xe` driver + Level Zero 1.28.2 stack P2P-capable but `ze_peer` unbuilt,
and flagged that O1 needed a dedicated window plus the D2 swap decision. Both cleared

receipt `br-20260718-054624-4d91410a`), and the window opened with both cards idle.

* **Host:** AM4 (Ryzen 9, 32GB DDR4), native Ubuntu, `xe` kernel driver, Level Zero 1.28.2
* **GPUs:** 2× Intel Arc Pro B70 (device id `0xe223`, 2800 MHz)
* **Occupancy at bench time:** `b70-planner`/`b70-critic` both already `inactive`, zero GPU
  processes — nothing was stopped, nothing needed restoring.

## 2. Method

`level-zero-tests` @ `26e0fab`, built entirely without sudo:

1. Repo top-level hard-requires libpng → local `libpng` 1.6.50 built/installed under
   `~/o1/prefix`, passed via `-DCMAKE_PREFIX_PATH`.
2. `-DBUILD_ZE_PERF_TESTS_ONLY=ON` (skips the gtest/conformance side entirely);
   boost absent → only `ze_nano`/`ze_image_copy` skipped, `ze_peer` unaffected.
3. `-DCMAKE_POLICY_VERSION_MINIMUM=3.5` shim for CMake 4.2.3.
4. Binary: `~/o1/level-zero-tests/build/perf_tests/ze_peer/ze_peer`.

Engine query (`-q`): each B70 exposes group 0 (compute+copy, `-u 0`) and group 1
(dedicated copy/blitter, `-u 1`), one queue each.

Three sweeps, 50 iterations, sizes 8 B → 256 MB:

* default — unidirectional bandwidth + latency, write + read, all device pairs (compute engine)
* `-b` — bidirectional
* `-t transfer_bw -u 1` — unidirectional bandwidth on the dedicated copy engine

## 4. Findings

1. **Perfectly symmetric.** 0↔1 matches 1↔0 to the second decimal in every mode — no slot
   ordering or device-index bias.
2. **Full wire speed — no P2P penalty.** The 14.3 GB/s write / 13.1 GB/s read plateau is the
   practical ceiling of PCIe 4.0 x8 (15.75 GB/s raw), consistent with the AM4 x16 bifurcated
   x8/x8. P2P through the root complex loses nothing.
3. **Root-complex contention in bidirectional mode.** Aggregates of 16.6–16.8 (write) /
   20.9 (read) GB/s, not 2× the unidirectional rate (~26–28) — concurrent bilateral traffic
   contends at the host crossbar. Reads overlap notably better than writes.
4. **The copy engine is a free transporter.** `-u 1` matches the compute engine to two
   decimals. P2P movement (weights, KV) can ride the blitter without robbing compute —
   route bulk transfers over the copy engine in any multi-GPU serving layout.
5. **Small transfers are latency-dominated.** Floor ~6.1 µs write / ~7.8 µs read. A 16 KB
   tensor-parallel all-reduce payload rides at ~6.8 µs — essentially the floor. Bandwidth
   only becomes the story ≥1 MB (the ramp is ≥92% of plateau by 1 MB), i.e. prefill and
   weight/KV bulk movement.
