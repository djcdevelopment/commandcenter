Capability Engineering — Roadmap

An organization that learns. Not an orchestration framework — the constructs are substrate-agnostic: nothing changes whether the builder is Claude, GPT, Ollama, a human engineer, or a test rig. The business of the organization is continuously engineering its own capability: observe → characterize → protect → challenge → expand → re-characterize.

Prime directive: the system describes itself by observing itself, not by declaring itself. Every construct below is either an event or a projection.

The Constitution

Clause 1 — No organizational truth may be authored if it can be derived.

Beliefs — findings, confidence, capability, qualification, mission requirements, competency, coverage — are projections over evidence. The moment any of them can be written by hand, the rot of every hand-maintained capability matrix begins. Anything that is not an event is a projection.

Clause 2 — No authored object may masquerade as a truth.

Authored objects are statements of will, never of fact: intent, accepted risk, suspensions — always with a reason, always in the audit trail. The stage-4 template: experiment plan authored at dispatch; experiment result derived, never authored.

Enforceability — determinism is the load-bearing property (D18).

Derived truth is only safe to never store as authoritative because projections are deterministic and re-runnable: stable IDs, no wall-clock stamps, diff-clean re-projection, thresholds as traced constants. A non-reproducible derivation engine is authorship with extra steps.

Residue — the one declaration that survives: what to observe.

Blind spots cannot discover themselves. The instrumentation decision (“observe this”) remains a human, authored act — Derek's role. Everything downstream of the sensor is derived. Coverage itself becomes a derived quantity in Stage 8.

S6

Qualification decay — preventative maintenance

DONE 2026-07-02

Shipped

qualification_run experiment type live; staleness is measured against the evidence watermark (the newest fact in the corpus), never wall clock, so re-projection stays diff-clean. Full decay/renew cycle proven in tests: 18 evidence-days stale → requalification_due → qualification candidate proposed (any qualified resource may run it) → one fresh success → qualified, demand withdrawn. Window: 7 evidence-days (D18 traced constant). A fleet-live cycle still awaits a real dispatch, per the real-gates-on-real-evidence restraint.

Goal

Capabilities expire like instrument calibrations. last_validated + re-qualification interval → stale capability with dependent work generates experiment demand. Don't wait for the spindle to snap; periodically verify it is still qualified.

Deliverables

  • New experiment type: qualification_run — motivating belief = “capability C still holds”
  • project_experiments.py synthesizes qualification candidates from capability staleness
  • Re-qualification intervals as traced constants (D18)

Definition of done

  • The loop closes on existing objects, no new machinery: capability → expiry → qualification experiment → evidence → association → capability renewed
  • One real capability driven through a full decay/renew cycle on the fleet

Why it matters

This turns the Experiment layer from opportunistic R&D into the org's protect-capability function — the maintenance half of capability engineering.

S7

Capability-directed dispatch

DONE 2026-07-02

Shipped

scheduler-decision.v1 carries capability_influence (required / matched / missing, per-candidate) beside policy_influence; requirements are derived from trace history — capabilities whose invariant matches the workload's task_kind. Proven in the capability-directed scenario: the qualified ollama candidate (base 0.8 + 0.15 match bonus) outranks the unqualified 0.9 — qualified capabilities are scheduled, not machines, and the decision says so. A stale qualification or out-of-envelope context does not match: untested ground is not qualified ground.

Goal

The scheduler stops asking “which builder?” and asks “which qualified capabilities does this mission require, and which resource combination satisfies them?” The aerospace / fab model: schedule qualified capabilities, not machines.

Deliverables

  • Mission→capability requirements derived from trace history (which capabilities similar past dispatches actually consumed) — never hand-authored, or the rot moves one level up
  • scheduler-decision.v1 extended: capabilities_required / capabilities_matched alongside policy_influence

Definition of done

  • Every dispatch explains its capability match the way it already explains policy influence
  • Requirement inference carries the same believed/observed treatment as everything else

Note

Requirement inference is itself a capability the org has to earn — expect it to start uncertain and graduate through its own findings.

Deliverables

  • project_coverage.py — blind-spot projection
  • Coverage gaps feed experiment candidate synthesis (the challenge-capability function)

  • “No silent caps” applies to the org describing itself: anything not covered is listed as not covered, never absent-and-implied-fine

Residue

What to observe remains the authored human act. Coverage tells Derek where his next “observe this” declaration will pay most.

S9

Competency — comparative excellence

EMERGENT — DO NOT BUILD

What it is

Capability answers can we do this? Competency answers how well, versus alternatives? — areas of excellence, relative strengths and weaknesses. What organizations actually reason about.

Why not build it

Let it emerge exactly as Capability is emerging: don't invent organizational truths, build derivation engines and let the constructs fall out. Its derivation source already runs daily: the competing-builders A/B (frontier vs AM4) and the planner ensemble produce pairwise outcomes on shared task classes.

Property to record from day one

Capability decays with time; competency decays when the comparison pool moves. “Excellent at long-context planning” can become false overnight because a frontier model updated, with zero change in our system. Competency statements must carry their reference set, the way a benchmark carries its baseline. This falls out naturally if competency is derived from comparative findings — one more reason never to hand-build it.

S10

Mission & Strategy — authored intent over derived capability

HORIZON

The reordering

Original roadmap said Policy → Strategy → Mission. It is now Capability → Mission → Strategy → Execution: every strategy is bounded by capability, every mission is bounded by capability, every experiment exists to change capability. Capability is first-class.

Shape

  • Mission = authored intent (Clause 2: will, with reason, audited) — bounded by derived capability
  • Strategy = emergent, like competency — expect it to be derivable from which capability investments actually paid off
  • Original stage-6 autonomy lands here: policy replay, counterfactual scheduling, experiment recommendation — now capability-directed

Gate

Do not design this layer until S5–S7 have produced real, fleet-derived capabilities. Missions authored against imagined capabilities are declarations — Constitution violation by construction.
