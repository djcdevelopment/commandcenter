Getting the Ledger — the Tool-Call Manifest

Every tool call through the HEARTH gateway becomes exactly one hearth-event.v1 line on an append-only ledger. This is the complete manifest of what has gone through the door. This doc is the how-to for reading it — where it lives, its shape, and every way to pull it.

Source of truth: events.ndjson Index: index.sqlite Contract: hearth-event.v1 Code: hearth/kernel/ledger.py
Where it lives. The ledger sits at $HEARTH_ROOT/var/ledger/ when HEARTH_ROOT is set, otherwise <repo>/hearth/var/ledger/ (gitignored). The live one is on OMEN, where the always-on gateway runs — not on the Windows box. Run any of the commands below on OMEN (or against a copied-down ledger dir).

The two files

events.ndjson source of truth

One JSON object per line, one line per tool call. This is the manifest. Append-only — there is no update or delete, by API design, not just convention. To eyeball it, just tail it.

index.sqlite derived / rebuildable

Only an index into the NDJSON: per event it stores a byte offset + length plus fast-filter columns (ts, caller_id, runner_class, tool, ok, duration_ms, task_class, model). query() uses it to seek and read the full event back from disk. If it's ever lost or stale, rebuild it from the NDJSON — the NDJSON never depends on it.

What one event holds

Built by new_event() in hearth/kernel/ledger.py. Note it stores digests + a 400-char args preview, not full args/results — a provenance manifest, not a payload archive.

FieldMeaning
event_iduuid4
tsISO-8601 UTC, Z suffix
caller{id, runner_class, node} — runner_class ∈ frontier | local | human

How to get it — four ways

1 · Raw — read the NDJSON directly

No Python needed; the file is the record.

# on OMEN
cd "$HEARTH_ROOT/var/ledger"   # or <repo>/hearth/var/ledger
tail -n 20 events.ndjson | jq .          # last 20 calls, pretty
jq -r 'select(.ok==false) | .tool' events.ndjson | sort | uniq -c   # failures by tool
wc -l events.ndjson                       # total calls ever

2 · Programmatic — the Ledger API

Reads full events back through the index, oldest first. Filters: caller (matches caller.id), tool, since (ISO-8601 UTC lower bound, inclusive), ok.

# from the repo root, in the fleet venv
python - <<'PY'
from hearth.kernel.ledger import Ledger
led = Ledger()                              # respects HEARTH_ROOT
print("dir:", led.dir)
rows = led.query(tool="local_generate", since="2026-07-09T00:00:00Z")
print(len(rows), "events")
for e in rows[-5:]:
    print(e["ts"], e["caller"]["id"], e["tool"], e["ok"], e["duration_ms"])
PY

3 · SQLite — fast aggregates without loading events

Query the index columns directly for counts/rollups. (Payload isn't here — join back to the NDJSON via offset/length, or use the API, when you need args/result.)

sqlite3 "$HEARTH_ROOT/var/ledger/index.sqlite" \
  "SELECT tool, count(*), round(avg(duration_ms)) ms
   FROM events GROUP BY tool ORDER BY 2 DESC;"

4 · Through the door — projections

The gateway's query_* tools and the projection layer (hearth/projection/ledger_adapter.py, economics.py, capacity.py) all read back through this same ledger — use them when you want the rolled-up belief/economics/capacity views rather than raw events.

Maintenance — reindex & verify

The CLI in hearth/kernel/ledger.py rebuilds or checks the index. Both respect HEARTH_ROOT and print the resolved ledger dir.

python -m hearth.kernel.ledger --verify    # read-only: does index.sqlite match events.ndjson?
python -m hearth.kernel.ledger --reindex   # rebuild index.sqlite from the NDJSON (source of truth)
  • verify → {ok, index_rows, ndjson_lines, mismatches}. A mismatch = an indexed slice that won't parse or whose event_id disagrees with its row. Mutates nothing.
  • reindex drops & recreates the table, re-streams the NDJSON in binary (byte offsets), and reinserts one row per line. A truncated final line is treated as an in-flight write and skipped; a torn line anywhere else is corruption and raises.
