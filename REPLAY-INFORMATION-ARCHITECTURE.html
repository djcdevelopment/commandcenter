Replay & Observability — Information Architecture

"Watch the organization think" is not a new belief layer — it is a view over the S1–S8 spine and the Δ2 physical overlay, both already contracted. This document does not design pixels. It decides what a replay shows, in what order of prominence, and where the line falls between rendering an existing derived field and inventing a new one. Two new derived quantities are proposed; everything else is disciplined rendering of contracts that already exist.

Constitutional constraint carried forward: a replay derives everything it shows. If a replay needs a fact no projection currently emits, that is a gap to close in the projector — never a value to hand-author into the replay layer itself.

1 — What Belongs on the Timeline

Three distinct kinds of thing share one time axis, and they must stay visually and structurally distinct because they answer different questions:

Authored spinesparse, high-signal→ Derived overlayre-projected, not stored→ Telemetry bandcontinuous, sampled
LaneContentsNature
Authored spine Mission accepted, experiment plan authored, policy override authored, dispatch decision, observe-this declaration. The events a human or the world actually committed. Discrete, sparse, never re-derived — these are the ground truth events.jsonl already is.
Derived overlay Capacity observations, findings synthesized/updated, associations formed, capability confidence changes, qualification transitions, coverage gaps opened/closed. Computed on demand by re-running the existing projectors up to time T. Never stored as a separate replay log — storing it would violate Clause 1 a second time, just one layer up.
Telemetry band gpu_temp_c, power_w, fan_rpm, clock_mhz, model_residency (Δ2, once instrumented). Continuous time series, not discrete events — rendered as a curve/band beneath the two event lanes, on the same axis, never merged into them.

Consequence: the replay engine has no new storage. It takes a time window and a run/mission ID, slices events.jsonl + the capacity-observation stream, and re-runs the association/capability/coverage projectors bounded to that window. This is exactly the determinism property (D18) the roadmap already requires of every projector — replay is just a projector called with a time bound instead of "latest."

2 — Which Events Deserve Visual Prominence

Prominence must not be a function of frequency (routine successful observations would dominate) or of lane (physical telemetry samples are the most numerous events in the system and the least individually interesting). Prominence should be a function of belief delta — how much an event changed what the organization believed, or how far it diverged from what capability/policy predicted.

TierExamplesDefault visibility
Tier 1 — always prominent Dispatch decisions; qualification status transitions (qualified↔stale↔requalified); policy overrides (authored, exceptional by construction); any finding whose confidence crossed a threshold band; coverage gap opened or closed. Rendered as markers on the primary lane, always visible at any zoom level.
Tier 2 — visible on zoom / secondary row Capacity observations that matched prediction; association formation; experiment results that confirmed an existing belief without moving it. Collapsed into a density strip until the operator zooms into that window.
Tier 3 — background, query-only Raw telemetry samples; routine successful runs with zero belief delta; re-projection bookkeeping. Never rendered as discrete markers — only as the continuous band (telemetry) or on explicit drill-down.

This requires one new derived quantity: a belief-delta / surprise score per event — cheap to define (magnitude of confidence change, or divergence between predicted capability match and observed outcome) and, like everything else in the ontology, computed, never asserted. It is the one genuinely new construct this document proposes for prominence ranking.

Full association computation (which findings compressed into which statement).

Raw telemetry samples underlying a thermal-decay finding.

Historical decay math for a qualification (evidence-day counting, window arithmetic).

Rule of thumb: if a field already exists on a shipped contract specifically to explain a decision (scheduler-decision.v1's two influence objects, experiment-result.v1's belief_before/after), it is default-visible. If a field exists to support a decision (provenance, raw evidence, computation trace) it is drill-down — present, never absent, but not competing for attention in the first pass. This mirrors the roadmap's own authored/derived split: default view = what changed and why; drill-down = what was the evidence.

Coverage is the one category that must never be silently dropped at either level — a decision made with an unmeasured input has to say so in the default view, per S8's "no silent caps" rule applied to the org describing itself.

4 — Merging Physical Telemetry With Reasoning

The wind-tunnel doc already puts physical telemetry (Δ2) between dispatch and capacity observation in the spine — it is an input to belief, not decoration. The replay surface should therefore treat it as a parallel, time-synchronized lane with explicit causal links, not a bottom strip that happens to share an x-axis:

Two lanes, one axis

Reasoning lane (authored spine + derived overlay) above, physical lane (telemetry curve, phase-segmented) below, both addressed by the same timestamp. Nothing about the physical lane is folded into the reasoning lane's event stream — they stay separately inspectable so an operator can look at either in isolation.

Evidence links, not decoration

Wherever a derived finding or capacity observation actually consumed a telemetry window as evidence (e.g. a sustained-load-decay finding), a connector ties the reasoning-lane node to the physical-lane window it was computed from. No connector, no claimed relationship — this is provenance, rendered across lanes instead of within one.

Phase segmentation — the second new derived quantity

The physical lane is only useful for debugging if it is segmented into named phases (cold-load → warm-ramp → sustained → cooldown) rather than left as a raw curve. This segmentation is derivable from model_residency + clock/power trend, computed once per run, and is the second new construct this document proposes. It is what lets "why was it slow" decompose into phases instead of requiring the operator to eyeball a temperature graph.

5 — Answering the Operator's Questions

Each question below should resolve to a specific, already-designed (or minimally-extended) field — not a new investigation each time. If a question can't be answered this way today, that gap belongs in S8 coverage, not in a bespoke debugging script.

Why was this resource selected?

Click the dispatch node → render capability_influence (required / matched / missing, per candidate) beside policy_influence as a ranked comparison table, not a single winner. The scheduler already produces this; replay just has to stop collapsing it to one line.

Source: scheduler-decision.v1 (S7, shipped)

Why was it slow?

Decompose elapsed time across phase boundaries: queue/policy wait, cold-load, warm-ramp, sustained inference, cooldown, assay. Each phase is timestamped on the physical lane (via phase segmentation, §4) so slowness attributes to a specific phase instead of a single opaque duration.

Source: dispatch timestamps + new phase-segmentation derivation

What changed?

Diff mode: pick two points on the timeline, render the delta in findings / capabilities / coverage between them, sourced directly from belief_before / belief_after already required on every experiment result — no new field, just a two-point rendering mode instead of single-snapshot only.

Source: experiment-result.v1 (S4, shipped)

Why did confidence increase?

Click the finding/capability node → list the specific supporting findings that fed the computed confidence, their recency, and any contradictions counted against it. Provenance is already a definition-of-done requirement (no orphans) — this is that list, surfaced.

Source: capability.v1 confidence computation (S5b, shipped)

Was the bottleneck organizational or physical?

Read directly off the two-lane layout (§4): a long reasoning-lane span with a flat physical lane means organizational (queue, policy gate, requirement inference); a short reasoning-lane span with a physical lane showing thermal throttle or cold-load ramp means physical. No new synthesis is required — this question is answerable by construction once the two lanes share an axis and phase boundaries are marked, which is exactly why the two-lane structure in §4 is the load-bearing recommendation of this document.

Recommendations Summary

R1

Replay model

VIEW, no new storage

What

Replay = existing projectors called with a bounded time window against events.jsonl + the capacity-observation stream. No replay log, no snapshot table. If it can't be re-derived on demand, it isn't ready to be replayed.

Why

Preserves D18 (determinism) and Clause 1 (no authored truth where derivation exists) without exception. A stored replay log would be a second, driftable copy of belief.

R2

Timeline structure

three lanes, one axis

What

Authored spine (sparse markers) / derived overlay (density strip, expands on zoom) / physical telemetry (continuous, phase-segmented band) — sharing one time axis, never merged into each other, connected only by explicit evidence links where provenance says so.

Why

Keeps "what did we decide," "what did we come to believe," and "what did the hardware do" separately falsifiable — which is the precondition for the organizational-vs-physical bottleneck question in §5.

R3

Visualization hierarchy

1 new derived field

What

Default prominence = belief-delta / surprise score, computed per event, not chronological density or lane. Tier 1 (belief-moving) always on-screen; Tier 2 (confirmatory) collapses to a strip; Tier 3 (routine/raw) query-only.

Why

A replay that shows everything with equal weight is unreadable at fleet scale; a replay that shows only "interesting" events by editorial choice reintroduces authorship. A computed delta score keeps curation itself a projection.

R4

Debugging workflow

1 new derived field

What

Every one of the five stock operator questions resolves to: click a node → render a specific existing contract field (or the phase-segmentation field, the one other new construct) → done. No bespoke investigation path per question.

Why

If a question can't be answered this way, the gap is a coverage gap (S8), not a reason to hand-build a one-off debugging script — the same discipline the roadmap already applies everywhere else.

Cross-Reference — Shared Dependency With LEARNING-RATE-CRITIQUE.html

The derived-overlay lane (§1) and diff mode (R2/"what changed?") assume machinery that doesn't exist yet.

LEARNING-RATE-CRITIQUE.html (Q2, terms 2–4) independently identifies the same gap from the Δ3 metric side: association.v1, capability.v1, and coverage.v1 are current-state snapshots, not transition logs. "What associations formed / what qualifications renewed / what gaps closed between two points in time" is a deterministic diff between two projection runs — buildable within the constitution (the spine is replayable, so the diff is itself deterministic), but it is missing machinery, not an existing derived field, for exactly the three object types replay's derived-overlay lane needs to animate over a scrubbed window.

Only the belief_before/belief_after pair on experiment-result.v1 is diff-ready today (a materialized field pair on one record). Everything else in the derived-overlay lane is render-only at a single point in time until the snapshot-diff mechanism both documents independently converge on gets built once, shared by replay's diff mode and Δ3's associations/qualifications/coverage terms alike. Build it once — not twice under two names.

Constitution Check

The replay layer authors nothing.

Every lane, every prominence ranking, every answer to an operator question is a projection over events and existing derived contracts. The replay surface adds exactly two new derived fields — belief-delta/surprise score and phase segmentation — both computed, neither asserted.

What remains authored: nothing new.

No new "observe-this" declarations are required beyond Δ2's telemetry fields, already proposed. The replay layer is purely a rendering discipline over what Derek has already chosen to instrument.
