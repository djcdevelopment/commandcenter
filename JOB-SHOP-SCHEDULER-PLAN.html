Job Shop Scheduler — Build Plan

Created 2026-07-04 · Origin: Derek's 2019 article "Solving the Job Shop Problem" (OR-Tools / CP-SAT) declared the north star for HEARTH + mechnet + Watchfire. Landscape confirmed by 3 scout agents 2026-07-04. Executor tier: Sonnet agents (Fable briefs & reviews, per token-tier delegation).

The thesis in one paragraph

Classic JSP needs three inputs the lab has been quietly instrumenting: task durations, machine capabilities, and constraints. The HEARTH ledger measures durations per node; the assay earns capabilities; occupancy/fan-out/precedence are the constraints. The scheduler is advisory-first (shadow mode) and the objective is minimize metered tokens subject to deadline — not raw makespan — per the two-economies doctrine. The conductor stays the ONE scheduler; HEARTH only orders/filters the eligible-builders list (CCMETA) once the shadow assay proves value.

JS3 — CP-SAT shadow scheduler: hearth/scheduler/ + propose_schedule

Repo: C:\work\commandcenter. Advisory only — this slice must not change any dispatch behavior.

  • Add ortools to the hearth venv requirements (find where deps are declared for fleet-worker-node/.venv-omen; pin a version with cp312-win wheels). If install fails in your environment, code against the API and mark install as a handoff note — do NOT stub the solver with a fake.
  • New package hearth/scheduler/ with three pure modules:
    • ontology.py — dataclasses: Job (plan_id, task_class, precedence list, deadline_s|None, est_duration source), Machine (name, kind local|frontier, token_cost_weight, capabilities/tags, available bool), ScheduleProposal. Loaders: machines from fleet/inventory.toml + hearth/etc/backends.toml + knowledge/capacity.json (tolerate absent capacity → fall back to a declared default duration per task_class); jobs from a plain list-of-dicts snapshot (caller provides — do NOT ssh from this module).
    • solve.py — CP-SAT model: interval vars per (job × eligible machine), AddNoOverlap per machine, precedence constraints, optional deadline. Objective = W_TOKENS * Σ(est_tokens × machine.token_cost_weight) + W_SPAN * makespan with frontier machines carrying high token weight and local machines ~0 (two-economies objective). Durations = capacity p90 for the (task_class × machine) bucket, fallback chain p90→tool-bucket→default. Rolling-horizon by construction: solver is stateless, takes a snapshot, returns a proposal; 10s solver time limit.
    • decision.py — render a proposal into a scheduler-decision.v1 record (READ contracts/scheduler-decision.v1.schema.json and conform: candidates_considered, selected, decision_reason, predictions, economy_influence). Validate against the schema in tests.
  • New toolsurface module hearth/toolsurface/scheduler.py with propose_schedule(jobs_snapshot: list|None = None): if no snapshot given, build one from local runs/ info available WITHOUT ssh (or accept that live snapshotting lands in JS4) — the tool must work fully offline on a provided snapshot. Returns the proposal + decision record; the gateway wrapper ledgers it automatically. Export via get_tools(); register the provider in the gateway --providers default list (find where the provider spec is declared, e.g. the start script hearth/etc/start-hearth-gateway.cmd).
  • Tests: deterministic 3-job × 3-machine fixtures — assert no-overlap holds, a frontier-only-capable job goes to frontier, a both-capable job goes LOCAL when deadline is slack (token objective dominates) and FRONTIER when deadline is tight (span term dominates), decision record validates against scheduler-decision.v1. If ortools import is unavailable in CI env, skip solver tests with a clear skip reason but keep ontology/decision tests running. NO SSH/network in tests.
  • Deliverable: passing suite + a worked example in the summary (3 jobs, the schedule CP-SAT chose, and why).
