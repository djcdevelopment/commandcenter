Signed off
Host: OMEN Hyper-VConductor: cc-conductor (Ubuntu)Net: Tailscale

Why

Every blocker so far was the Windows↔Linux seam (path mangling, BOM, mcp-over-ssh on Windows) plus Default-Switch IP fragility. Root-cause fix: the conductor becomes a native Linux VM on OMEN running Claude Code, and the fleet moves onto Tailscale for stable addressing. No Windows in the dev or data path.

Steps

  • Clone the proven Ubuntu image → cc-conductor; dedup identity (hostname / machine-id / ssh host keys).
  • Install Claude Code; bring in the commandcenter repo & memory.
  • Join cc-conductor (and the worker VM) to Tailscale; retire fragile Default-Switch IPs.

Sign-off =

  • Claude Code runs inside cc-conductor; repo + memory present.
  • cc-conductor reaches the worker VM, AM4, i5 by tailnet name.
  • You drive it via SSH / VS Code Remote-SSH — Windows seam gone.

Constraint & Control Points (cross-cutting)

Architecture invariants

  • Local-first: prove every contract on the OMEN VM before distributing. Distribution = a placement change, not a rewrite.
  • Plane separation: MCP = control surface, never the data bus.
  • Location transparency: cross-host = MCP/NATS, never filesystem. No new SSHFS coupling.
  • Idempotent triggers; reuse am4-fleet-node, deltas only.

Operating constraints

  • AM4 image gen running: B70s running image generation with watchdogs + monitoring. MCP control calls (node_status, etc.) clear. Coordinate before GPU-contending ops.
  • OMEN thin NIC: push heavy external I/O to the i5 edge, not through OMEN.
  • Identity (your edge): token/key auth, secrets on-node; fleet-wide auth model (keys vs Tailscale SSH) decided at S4.
  • No overcommit: commit S0–S1 only; roadmap stays uncommitted. fx99 parked.
