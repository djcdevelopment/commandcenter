What HEARTH is

HEARTH is the single always-on MCP gateway that fronts every offload in the lab — local-model calls, fleet task dispatch, file/git/test operations, health checks, knowledge queries, and scheduling — so that every crossing lands on one ledger. It runs on OMEN at 127.0.0.1:8710 (FastMCP over streamable-http), started by a login/scheduled task and kept alive by its own guard dog. Three kinds of callers reach it: frontier Claude Code sessions (this one included — native mcp__hearth__* tools via the repo's .mcp.json), the local model (qwen3-coder:30b, driven through a tool-calling loop), and the fleet (builder VMs dispatched through the async task lane).
The doctrine: ADR-0005, “one boundary, three planes”. SENSE (AM4-MCP + OTel/Jaeger) and ACT (the mechnet: builders, VMs, backends) are separate planes — but every offload or engineering crossing between an agent and either plane goes through the one boundary: HEARTH. Not for tidiness — because at this stage of the lab, data is the deliverable. A second door that carries work is a second place observations can leak, and the belief layer (S1–S8) only learns from what actually reaches the ledger. The companion rule, for frontier sessions specifically (hearth/callers/frontier.md): act only through gateway tools — no SSH side-writes, no direct knowledge/ edits, no out-of-band git push. If a tool is missing, that's a gap to name, not a door to route around.

Architecture

The contract — what every call gets

Every provider tool is a plain typed Python function with a docstring; the gateway wraps each one so a caller can't get a partial contract. From hearth/kernel/gateway.py::make_wrapper:

  1. Identity resolved from the X-Hearth-Key header. Unknown or missing keys are rejected — and the rejection is itself a ledger event, recorded under a synthetic __unauthenticated__ identity (key fingerprinted, never logged raw).
  2. Guard-checked before the call runs: a knowledge-path guard (only the knowledge module's own tools may touch knowledge/*.json) and a fixture-taint guard (refuses args carrying fixture-derived source paths — the 2026-07-02 corpus-overwrite incident, designed out at the door).
  3. Timed — wall-clock duration in milliseconds, on every call.
  4. Digested, not recorded — sha256 of the JSON-serialized args and result, plus a 400-character preview. Audit truth, not a replay log (that's the workflow ledger's job — see below).
  5. Bucketed by task_class (inference, dispatch, test, io, health, vcs, query, …) for cost and capacity analytics.
  6. Ledgered, always — a hearth-event.v1 lands in the append-only kernel ledger even when the tool also writes its own domain event (record_event does exactly this; ADR-0011 is the one-file answer for why that's two facts, not one fact twice).

The guard dog — Watchfire

ADR-0007: the watchdog's sense was upgraded from liveness (is a service reachable?) to coherence (do the sources agree?). Every false signal in the session that motivated this was the same shape — two sources that should have agreed, and didn't, with no single view reconciling them.

The auto-heal boundary is deliberately narrow: AUTO_HEAL_KINDS = {phantom_in_flight}. Every heal is self-documenting (_healed_by / _healed_at) and reversible (delete the stub), and a heal must resolve a gap, never relabel it as a fresh crash. Everything ambiguous stays eyes-only, handed to a reasoner — the same "gate, not an oracle" discipline as the fleet assay (ADR-0001).

Cadence — two ticks, one engine

The newest slice (commit 7428a94): a cheap patrol(refresh=False) snapshot runs every 5 minutes into a small rolling file (capped ~1 hour). The existing 15-minute watchdog tick reads back the last 3 snapshots and tags each gap persistent / new / resolved by plain set comparison on (kind, plan_id) — then runs the liveness heal pass and best-effort refreshes capacity, the AM4 catalog, and a 20-run hindsight replay. The coherence sweep never affects the liveness health verdict; it's additive and best-effort.

dream is the guard dog's off-duty spell: when nothing needs watching, it can render on AM4's idle GPU instead — the lab having some soul, thermally guarded at 95°C so it never contends with real work.

The scheduler — a job shop for GPU time

The newest layer, and a full-circle moment: Derek's 2019 article on rebuilding a manufacturing MES with OR-Tools turned out to be the destination all along. GPU model residency is machine setup time; a model swap is a changeover. ADR-0008 is the current gate.

Objective (two economies): minimize metered frontier tokens first, makespan second — local placement is always preferred unless a hard deadline forces parallelism. Constraints: one machine per job, no-overlap per machine, precedence, hard deadlines, plus the setup-aware layer (JS7b): model residency, a single shared DDR4 staging slot on AM4, and a per-card VRAM budget.

The imagegen-250 assay

MakespanModel loadsSetup timeDeadline misses
FIFO baseline3378.0s24410.0s19

CP-SAT scheduler3124.0s3 50.0s0 OPTIMAL

250 synthetic image-gen jobs across sd3.5_large / sdxl-base / flux-schnell on a single stateful AM4 machine. The solver matches the smart baseline's model-load count structurally (by packing residency) while clearing every deadline miss — 88% less setup time than naive FIFO.

Advisory-only, on purpose. propose_schedule and schedule_hindsight are read-only — nothing dispatches from them. The one-scheduler doctrine holds: the conductor owns dispatch; HEARTH only ever names eligible builders. JS5 (scheduler-informed ordering, behind HEARTH_SCHEDULER_ADVISE=1) is specified but deliberately unbuilt until the regret trend — accrued automatically every 15-minute patrol tick, on real runs — shows sustained, non-trivial savings. Today that trend shows zero token regret, because the dispatch history so far is all-local: the synthetic assay justifies the formulation, not yet the actuation.

Origin artifact

HEARTH's very first local dispatch (qwen3-coder, before the ledger had a single real rep) was asked, in effect, to describe the system it was running inside. It confabulated a generic textbook description of an operating-system kernel — not a single fact about this repo. It's kept verbatim, on purpose, as the system's oldest artifact: an honest first data point about local-model reliability, from hearth/var/first-dispatch-report.md. 
The HEARTH kernel serves as the core operating system component that orchestrates
hardware resources, manages system processes, and provides a stable environment for
applications to run. It handles tasks such as memory management, process scheduling,
and hardware abstraction.

For contrast, this page you're reading was assembled from the actual source, ADRs, and a fresh test-collection run — the whole reason the belief layer exists is to stop trusting the confident-sounding version and start trusting the ledgered one.
