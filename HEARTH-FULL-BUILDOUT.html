HEARTH — the always-on lab

Full build-out design · drafted 2026-07-03 · status: DRAFT — awaiting Derek's review · name proposed from the fire family: the hearth is the small fire that never goes out, at the center of the camp.

0 · The inversion, stated once

Today: frontier agents sit on top and reach down into machines (SSH, direct file edits); the lab observes what it can. HEARTH inverts this: the lab is the only thing with hands. Every actor — Claude, Codex, qwen3-coder, mixtral — is a stateless caller behind one always-on MCP surface. Building the lab and using the lab are the same instrumented act, so the lab learns about itself even while frontier models construct it.

Scope shrinks to match: HEARTH is the mini commandcenter built for already-specific work requests. It keeps one always-on gateway + one local inference worker warm on OMEN so the system can react quickly while still capturing communication, action, and learning. Everything else (AM4, VM farm) becomes capacity the lab summons through its own tools.

1 · Layer map

L0 KERNEL Frozen core — changes by ceremony only

  • MCP gateway — the single always-on daemon, living on OMEN. All tool traffic enters here. It terminates caller auth, stamps provenance, and fans out to tool providers (local or remote via SSH-stdio, the existing worker-mcp-server pattern).
  • Event ledger — append-only store (NDJSON + SQLite index). Every tool call = one event: caller identity, runner class (frontier|local), tool, args digest, result digest, wall-time, token/watt cost fields. This is the raw ore the belief spine smelts.
  • Guard stack — corpus_guard + fixture-taint guard (both shipped) + watermark-monotonicity guard (proposed in Three Chairs addendum). Guards run in the gateway, not in agent convention.
  • Identity & auth — per-caller keys. A caller is (who) × (runner class) × (node). Frontier sessions get MCP credentials only; they are locked down to the gateway surface and do not receive general SSH write access.
  • Kernel change path — kernel repo is separate (or a protected subtree); changes require a signed-off dispatch through a dedicated kernel_change tool that snapshots before/after. A break-glass path exists as explicit operator policy for incidents, but it is not part of the normal build path and is not implemented in HEARTH yet. The bootstrap loop (capture layer edited through the capture layer) is closed by making kernel edits rare, logged, and reversible.

L2 MUSCLE Compute — callers and workers, all behind the same door

  • Standing (always-on): OMEN hosts the gateway and omen-worker-1 (qwen3-coder:30b, 54.6 tok/s) as the resident local brain. Ollama moves from login-start to service — with HEARTH capturing everything, always-on is safe by design, and routine local development actions stay warm instead of waiting on summoned external capacity.
  • Summonable: AM4/homebase (hermes/vllama aliases, Arc pool, ComfyUI), VM farm builders, claudefarm1 assay. They register capabilities on wake (self-announcing MCP servers — pattern already built).
  • Callers: frontier (Claude Code, Codex pours) and local runners share one client contract — runner.json grows an mcp_endpoint field and the agent-agnostic design carries over unchanged.

L3 MIND Learning loop — the lab as its own subject

  • Projection: S1–S8 spine consumes the unified event ledger instead of scattered run events. Capabilities are re-earned automatically — the corpus-overwrite wound heals as a side effect of normal work, and can't reopen (single writer + guards).
  • Assay: dynamic assay stays the regression gate; the new stream-scoped acceptance assay (the pour's finding) grades dispatches flowing through the gateway.
  • Economics ledger: every event carries cost — metered tokens (frontier) or watt-seconds/tok-s (local). Knowledge-per-local-hour becomes a computed metric, not a doctrine sentence.
  • Routing policy (the payoff): conductor learns dispatch = argmax(capability × grade ÷ cost) per task class. This is the offload engine — Derek describes an effect, the ledger finds the math, the router applies it. Frontier gets reserved for planning & synthesis (the top box of the sketch), mechanical middle moves to electricity.
  • Naadam for free: frontier vs local on identical tools with identical provenance IS the proving-ground comparison arena; the multi-tenant/BYOK version for the Edgeless colleague is this ledger with CIAM in front.

L4 FACE Interfaces

  • P0 idea pipeline (speak→board) stays the human front door; its output becomes dispatch_task calls.
  • Living HTML plans on :8080, now renderable from ledger state (real numbers, not hand-updated prose — fixes the "worth is authored" projector-prose class of drift).
  • OTel→Jaeger on AM4 unchanged; gateway spans slot into the existing pipeline.

4 · Build-values check

  • Visibility — strongest version yet: one door, every act a ledger event, plans rendered from state.
  • Testability — gateway is a choke point: replayable (mission replay doctrine — re-run a day's ledger against a changed lab), guards unit-testable, tool contracts assayable.
  • Resilience — kernel is small and boring; workers stateless and summonable; ledger append-only; the failure mode "agent silently mutated state" is designed out rather than patrolled.

5 · Decisions on Derek's desk

  1. Where the kernel lives: decided — HEARTH lives on OMEN.
  2. How hard the lockdown: decided — frontier callers are locked down to MCP-only operation. Common local development actions should run locally through HEARTH without depending on external agents.
  3. Kernel ceremony: HEARTH is the mini commandcenter for already-specific work requests; kernel scope is the HEARTH gateway/ledger/guards/auth surface, and break-glass exists as operator policy but is not implemented in this slice yet.
  4. Ollama always-on: locked in on OMEN. omen-worker-1 moves from login-start to service; HEARTH captures the economics and usage trail directly.
  5. Name: HEARTH proposed (fire family: the small always-burning fire at the center). Veto freely.

6 · Risks, honestly

  • Tool-surface friction: if a needed tool is missing, frontier agents will route around. Mitigation: H1 parity gate + a request_tool escape hatch that logs the gap instead of hiding it.
  • Gateway as single point of failure: accepted deliberately — it's also the single point of truth. Ledger is append-only on disk; gateway restart loses nothing.
  • Local tool-calling quality: qwen3-coder is decent at tools, mixtral weaker; H3's graded-task gate measures this instead of assuming it.
  • Latency tax: every op through one daemon adds hops. Coarse digest tools (L1) more than repay it in frontier round-trips saved.
