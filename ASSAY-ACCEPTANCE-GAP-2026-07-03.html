The Assay Acceptance Gap

Codex poured the five-stream worklist (A1, A2, B1, B2, C1) through the fleet. Two landed; three did not. This documents what the pour actually measured — not a failure of the worklist, but a diagnosable defect in the fleet's scorer. The assay is a regression gate ("did you break the baseline?"), not an acceptance oracle ("did you do what the stream asked?"). Grounded in POUR-LEARNINGS-2026-07-03.md and POUR-STATUS.md, cross-checked against the fetched winner branches.

Author role: instrument review, not dispatch Status date: 2026-07-03 Verdict: stale-base solved; branch-quality selection is the live bottleneck
Headline: the pour worked as an instrument. It stress-tested the scorer and the scorer failed in one clean way — a branch that does nothing passes the assay. Stream A2's winning branch's only meaningful diff was retro.md, yet it won on 130/130.

1 · The pour was an experiment on the instrument

The worklist was the payload; the fleet was the subject. Two results, cleanly separated:

Solved — the stale-base problem (infrastructure)

The first bad wave built from the conductor's stale ontology mirror (~/work/commandcenter-ontology/farmer-repo), not local master. Pushing master → mirror → resync fixed it, proven by C1 r4 where both builders independently reached 139/139. This is durable and no longer a concern.

Discovered — the assay cannot enforce stream-scoped correctness (the finding)

The winning assay for every landed run reads pytest: 130/130 passed; imports: 0/0 ok; score=70; isolation=docker. That grades only whether the branch left the existing baseline intact. It never asks whether the branch produced the artifacts the stream demanded. Those are different questions, and the pour is where the difference became load-bearing.

2 · The null-action exploit

When a stream's deliverable is "add new code and new tests," a branch that adds nothing scores identically to one that succeeds — because the fitness function measures don't-break, never did-do. An empty branch keeps all 130 tests green, imports clean, isolation intact: a perfect score for having done no work. That is not a hypothetical — it is exactly how A2's winner was selected. The irony writes itself: A2's own mission is to stop a re-projection that sees less evidence from silently overwriting the knowledge store. The assay did the same shape of thing to itself — it accepted a no-op as a win.

Read together: the acceptance gap is universal. It was invisible on B2/A1 because their deliverables happened to overlap what the assay can see; it was glaring on A2/B1/C1 because theirs did not.

4 · Why this is the bottleneck, not dispatch

The lab's whole shape is competing build-outs → score → learn, laps on electricity. The scorer is the fitness function of that loop. A scorer that can't tell "did the task" from "didn't break anything" forces a human curator back into every landing — which is precisely the bottleneck the system exists to remove. Codex's own read lands here: "the next bottleneck is branch quality selection, not raw dispatch." The recommended workaround — curate every stream from the approved prompt, treat the fleet as a draft generator — gets the work done but quietly re-inserts the human as the rate limiter. The finding is that the next increment worth building is the scorer, because that is what lets the fleet auto-land instead of draft.

5 · The fix — a stream-scoped acceptance assay

Score = regression-gate AND acceptance-manifest AND scope-fence

The acceptance contract already exists, latent, in every stream prompt's DEFINITION OF DONE: required new files, required grep targets, named new test files, an expected baseline delta, an out-of-scope list. Nothing new needs to be authored — the intake already states it in prose. The increment is to make that contract machine-checkable per dispatch and require it in addition to the baseline gate.

Latent in the stream todayMachine check the assay should runWhich failure it would have caught
DoD names new files (e.g. corpus_guard.py) Those paths exist and are non-empty in the branch diff. A2 null-action (no new file) → cannot win.
DoD lists grep targets (the six B1 amendment strings) Each target string is present in the named file. B1 destructive rewrite (targets absent / file mangled) → cannot win.
DoD names new test files and an expected count delta Named tests exist, pass, and baseline count strictly increases by the expected amount. C1 r3 (failing tests; no real delta) → cannot win.
OUT OF SCOPE list (files the stream must not touch) The diff touches only allowed paths; out-of-scope edits fail the branch. B1 rewriting whole docs, A2's unsolicited knowledge/* edits → fenced out.

This keeps the assay honest about its own constitution: it authors no new belief — it just executes the acceptance contract the stream already declared, the same way a test executes an expectation. A null-action branch fails the manifest by construction; a destructive rewrite fails the scope fence; a green-but-wrong branch fails the delta. The regression gate stays necessary; it stops being sufficient.
