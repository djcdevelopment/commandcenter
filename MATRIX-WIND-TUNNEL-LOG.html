Matrix Wind-Tunnel — Lab Log

Living document · started 2026-07-05 · the mechnet's planning-quality experiment: does self-refinement actually improve planning, and does it depend on the model / hardware / task? Two-economies doctrine — laps on electricity across AM4 (2× Intel B70) + OMEN (RTX 5070). Updated as runs land.

What we built (the instrument)

Commander intent lane LIVE

A local author↔critic refine loop that takes frontier out of the run loop (ADR-0012). hearth/commander/ + door tools refine_idea/refine_result + CLI. This loop is the cell engine of the matrix.

Matrix dataset harness LIVE

Per-role backend routing in run_refine (planner and critic on different boxes) + hearth/experiments/ (cells, held-out critic-panel scorer, dataset). Each cell = (planner × critic × prompt × laps × ordering) → a scored proposal. Fully offline-testable (237 toolsurface tests green); the AM4/OMEN legs run live.

The experiment

Finding 1 — refinement is not a free win for planning PRELIMINARY · superseded by Round 4 (2 laps wins)

The rounds below (3 prompts, low n) suggested laps only hurt. The 6-prompt n=12 Round 4 sweep corrected this: one review pass helps; only over-refinement hurts. Kept for the honest arc.

Across the 12-cell pilot (2026-07-06), 3 laps scored lower than 1 lap for both planners. More refinement, on aggregate, made planning answers worse, not better.

Finding 2 — the effect is task-dependent (but never positive) REVISED by sweep

Revised by the n=12 sweep: the pilot suggested refinement helped escalate-or-not; the sweep shows that was noise (flat at 84.2). Corrected read: refinement ranges from harmless (escalate-or-not, flat) to harmful (choose-next-agent, −12.5 at L3) — never a net win. Pilot n=1 numbers below, kept for the record; the authoritative per-prompt×lap view is in the Confirmation sweep.

escalate-or-not83.588.5refinement HELPED (+5)

Read: the skeptical critic pushes toward more complexity/verbosity; for a crisp routing decision (choose-next-agent) or a deliberately-terse skeleton that penalizes; for a judgment call with tradeoffs to surface (escalate-or-not) it pays off.

Finding 3 — the critic never converged pilot

converged=False on all 12 cells. The critic never emitted VERDICT: CONVERGED — it always found something to revise, so every cell burned its full lap budget. "Refine until converged" would run forever here; the lap cap is what stops it. The convergence bar (or critic prompt) needs a look — and it means the lap count, not convergence, is the real control knob.

Round 4 — the larger matrix flips it: 2 laps is the sweet spot AUTHORITATIVE · 48/48, 6 prompts, n=12/lap

Doubling the task coverage (6 planning archetypes) and the samples (n=12/lap), memory-safe cross-machine (AM4 30B planner ↔ OMEN critic), overturned the preliminary finding. With only the 3 original prompts and low n, refinement looked monotonically harmful. It isn't — the curve is an inverted-U with a peak at 2 laps:

Why the reversal: the 3 original prompts happened to be ones a review pass didn't help; the 3 new archetypes (resource-allocation, risk-triage, rollback-decision) benefit from one. And n=12 beats the earlier n=1–3 noise. Caveat: per-prompt cells are still n=2 (noisy — choose-next-agent swings 65↔83); trust the pooled by-laps (n=12). The honest arc — preliminary "1 lap" → authoritative "2 laps" — is the wind-tunnel working as intended.

Hardware finding — 32GB DDR4 is the binding constraint, not VRAM confirmed

The dedicated-loop run surfaced an infra truth flagged from the very start: the B70s hold 64GB VRAM total, but AM4 has only 32GB DDR4 host RAM. SYCL llama-server keeps host-side KV + compute buffers even with -ngl 99; co-loading the 30B planner + 14B critic at generous contexts OOM-killed the planner mid-run (503s until systemd restarted it). Dropping the critic 32k→8k ctx freed ~7 GiB (used 23→16 GiB) and restored headroom. The old Windows vllama had a max_host_used_gb_preflight: 22.0 gate for exactly this — the Linux serving has none yet (a real follow-up). Runbook: B70-CARD-MANAGEMENT.md.

What this changes updated by Round 4

  • Set the commander lane planning default to rounds=2 — not 3 (over-refines, 82.6), not 1 (under-refines, 81.2). Two laps is the measured peak (86.2). The earlier "1 lap" call was premature (3 prompts, low n); the 6-prompt n=12 sweep is authoritative. Still planning-specific — a BUILD/refine-for-code sweep could peak elsewhere.
  • The critic loop earns its keep — once. A single review pass helps (L1→L2 +5); the problem was never "refinement," it was over-refinement (L3–L4). Cap laps at 2 for planning rather than dropping the loop. (Convergence still never fires — the lap cap is the real control.)
  • The matrix earned its keep — twice. First it flagged a counterintuitive effect; then a larger run corrected the effect (1→2 laps) before it hardened into a wrong default. Idle local hardware, self-funding knowledge — exactly the wind-tunnel's purpose.
