The experiment

Question: the commander lane defaults to rounds=3 refinement laps on the assumption more refinement = better. Is that true for planning tasks — and does it depend on which model plans vs critiques, and on the task?

Finding 1 — refinement is not a free win for planning pilot, n=1/cell

Across the 12-cell pilot (2026-07-06), 3 laps scored lower than 1 lap for both planners. More refinement, on aggregate, made planning answers worse, not better.

Finding 2 — but the effect is task-dependent pilot

Refinement helped one task and hurt the others — so "how many laps" should be a per-task call, not a global default.

Prompt1 lap3 lapsverdict
choose-next-agent80.065.0refinement HURT (−15)
escalate-or-not83.588.5refinement HELPED (+5)
plan-skeleton85.075.0refinement HURT (−10)

Read: the skeptical critic pushes toward more complexity/verbosity; for a crisp routing decision (choose-next-agent) or a deliberately-terse skeleton that penalizes; for a judgment call with tradeoffs to surface (escalate-or-not) it pays off.

Finding 3 — the critic never converged pilot

converged=False on all 12 cells. The critic never emitted VERDICT: CONVERGED — it always found something to revise, so every cell burned its full lap budget. "Refine until converged" would run forever here; the lap cap is what stops it. The convergence bar (or critic prompt) needs a look — and it means the lap count, not convergence, is the real control knob.

What this changes

  • Commander lane default. If confirmed, rounds=3 is the wrong default — planning should use fewer laps, or a per-task lap count, or convergence-gating that can actually fire.
  • The matrix earns its keep. First cross-machine run already produced a counterintuitive, actionable result — exactly the wind-tunnel's purpose (knowledge-per-local-hour, on idle electricity).
