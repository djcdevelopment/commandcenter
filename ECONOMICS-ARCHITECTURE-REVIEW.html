The correction, in Derek's words

"The scheduler does not ask whether a resource is metered or sunk. It derives the current economic context from authored resource facts and live operating signals, then selects an optimization objective for this dispatch and explains that selection through economy_influence." — Derek, confirming the review's recommendation, 2026-07-02

This resolves the core objection below: economy membership is a per-dispatch derivation, not a static tag on a resource.

1 · Evaluate the two-economies model

StrengthsWeaknesses
Splits genuinely different objective functions (cost-per-outcome vs knowledge-per-hour); Δ3 correctly refuses to let raw GPU utilization stand in for the real objective — a hot GPU producing no belief change scores zero; reuses existing spine mechanics (policy_influence-shaped scoring), cheap to add; Δ4 idle-drain as an authored, suspendable policy is consistent with the constitution. Rests on "sunk compute costs ~nothing marginal," which is false under variable electricity price, battery power, and nonlinear wear — Δ2 senses power_w/thermals but nothing in Δ3/Δ4 consumes it; the model has no home for a single dispatch that spans both economies (local+cloud overflow).

2 · Is cost_class the right abstraction?

No — rejected as originally specified.

Three problems: it conflates ownership with objective (owning hardware doesn't make every dispatch on it economy-2 — a GPU serving an interactive session right now is scarce and latency-sensitive exactly like a rate-limited API); it violates Constitution Clause 1 ("no organizational truth may be authored if it can be derived") — whether to optimize for cost or knowledge-per-hour is derivable from observable signals, not a fact to hand-set; and it's one-dimensional where the domain has at least three independent axes (ownership, attendance, real-time price/power state).

Alternative consideredVerdict
optimization objectiveCorrect concept, but needs a mechanism to assign it — doesn't stand alone.
execution policyClosest fit. Matches how the spine already works — policy_influence is already "derived effect, explained, overridable." Fold economy selection into that same machinery instead of inventing a parallel concept.
resource classLegitimate as a narrower, genuinely-static field (resource.ownership) — but ownership ≠ economy; it's one input to the derivation, not the derivation itself.
operating modeLegitimate, needed as a separate derived signal (attended vs unattended) — this is what Δ4's "stops the moment he is using it" already implies but never named.
dispatch intentLegitimate, orthogonal — production vs characterization is a property of what the dispatch is for, not of which node runs it.

3 · Minimum ontology additions

Δ1′

Corrected — supersedes the original cost_class field

CONFIRMED

Authored (near-static facts)

  • resource.ownership: owned | leased | metered_provider — a real third class exists in the fleet today (farm VMs, cc-farmer tunnel: leased, neither owned-sunk nor pay-per-call)
  • resource.power_source: mains | battery | generator

Derived (per dispatch)

  • operating_mode: attended | unattended — from session-allocation events, never hand-set
  • nullable live signals: price/rate, battery level, rate-limit headroom — same "observe-this" pattern as Δ2's telemetry
  • optimization_objective — computed per dispatch from the above + dispatch intent; never stored on the resource

Explanation surface

economy_influence added to scheduler-decision.v1, parallel to policy_influence / capability_influence: records which signals were read, which objective was selected and why, and the counterfactual (what the other objective would have chosen and at what cost).

4 · How scheduler reasoning explains decisions

Cost-minimizing dispatchKnowledge-maximizing dispatch
Show the counterfactual against the next-cheapest qualified candidate: dollars saved, rate-limit headroom consumed. Show why this candidate beat the rest of the experiment queue: expected knowledge value (belief uncertainty × worth × recency), not just "was idle."

Mixed-economy dispatches (below) need both legs shown — a single scalar economy: sunk cannot represent a job that spilled from local to cloud mid-run.

5 · Hidden edge cases

CaseWhy it breaks the binary model
Mixed fleets / partial local / cloud overflow One dispatch, two economy legs (local does 80%, cloud finishes the hard 20%; or planner ensemble runs both concurrently for comparison). A per-resource tag has no home for this — argues for per-leg tracking and an unresolved credit-attribution rule when a finding only exists because both legs ran.
Changing electricity cost Falsifies "sunk ≈ free." Nothing in Δ1–Δ5 currently ingests a live price signal; Δ4 as written would idle-drain regardless of price.
Battery-powered systems Sharpest counterexample to ownership-as-economy: a laptop is "owned/sunk" by the model's logic, but a draining battery is scarce and depleting exactly like metered budget. Ownership says sunk; physics says metered.
Idle-but-warm models Model residency (already sensed by Δ2's model_residency) has real economic weight unused today: a warm-model match is near-free, a cold one pays load time + thermal ramp + possible eviction. Δ4's candidate scoring should prefer warm-model matches but currently doesn't.

Migration impact

Free now, expensive later.

Nothing built yet — this is the cheapest possible time to make the correction. If a static cost_class had shipped and Δ3/Δ4 were built against it, unwinding it after mission history was already projected against the wrong shape would be a second schema migration on a contract S7 (capability-directed dispatch) already depends on.

Unresolved questions

  1. Attention as a third economy? Is Derek's own wait-time meant to be modeled explicitly, or deliberately out of scope for now?
  2. Credit splitting. When a finding requires both a metered and a sunk leg (planner ensemble comparisons), how is knowledge-per-hour credited — split, duplicated, or attributed to neither individually?
  3. Leased resources. Farm VMs / cc-farmer tunnel are neither owned nor pay-per-call — what does resource.ownership: leased default to economically, and should that be observed rather than assumed?
  4. Δ4 preconditions. Does idle-drain need a live price/battery feed before it's safe to enable, or is a static "quiet hours" window an acceptable v1?
  5. Attendance threshold. Who/what authors "unattended" — idle timeout, explicit session-end event, or both? Wrong either way: false-idle burns electricity, missed-idle wastes real wind-tunnel time.
  6. Residency-aware scoring. Does model_residency fold into Δ3/Δ4 scoring now, or become its own later delta?
