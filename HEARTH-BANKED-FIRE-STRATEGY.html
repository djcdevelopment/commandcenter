Banked Fire

HEARTH idle-drain scheduling onto the mechnet — Δ4 made concrete  STRATEGY v0.1 — 2026-07-04

The ask (Derek, 2026-07-04): the dual B70s (32 GB VRAM each, 32 GB DDR4 system) are already on the mechnet. HEARTH should be able to use that hardware whenever no active mechnet job has it — research, simple builds, bigger inference — without hand-built configuration each time, with the mechnet healing itself. Every offloaded task = learning insight + hardware rep + frontier tokens saved.

The name: a banked fire is the hearth's overnight state — coals kept alive under ash so the fire rekindles instantly when needed. Idle GPUs holding a resident model, drained by queued work, yielding the moment real work arrives. Proposed codename: bankfire (Derek ratifies naming).

This is Δ4, not a new idea

TWO-ECONOMIES-WIND-TUNNEL.html already authored this delta: "When a sunk-economy node goes idle, the scheduler pulls the highest-value experiment candidate and runs it as a normal gated dispatch… The tunnel spins whenever Derek isn't using the machine — and stops the moment he is." The only new authored object Δ4 requires — the operating budget — was ratified with D1 economics on 2026-07-03 (fad3be9), and 46 experiment candidates are already priced. The policy layer exists. What's missing is plumbing, and less of it than expected.

Design principles

  1. One scheduler. The conductor owns queued/async work. HEARTH gets a door to it, never a rival queue. HEARTH's own routing decisions are confined to synchronous inference calls.
  2. One config file. Backends are declared once, declaratively; router, watchdog, and doorcheck all read the same declaration. No per-incident custom config — that's the "wasting time debugging" trap.
  3. Serve-truth occupancy. "Idle" is asked of the systems that know (conductor: active dispatches; backend: in-flight slots), never inferred from wall clocks or hope. vllama's core value, applied.
  4. Mechnet jobs always win. Opportunistic work runs on short leases and doesn't renew when real work appears. "Stops the moment he is" is a lease policy, not a kill signal.
  5. Respect the DDR4 ceiling. 32 GB system RAM makes model swaps the expensive operation. Policy: prefer the resident model; swap only when idle-time and queue depth justify it (hysteresis).
  6. Every dispatch is an assay observation. Ledger records backend, occupancy-at-decision, tokens, duration. The B70/Qwen3-30B combo is a second clean local combo — exactly what the G3 gate is waiting for. Banked-fire traffic re-earns beliefs as a side effect.

Architecture — two lanes through one door

Lane 1 · Inference (synchronous, seconds)

local_generate grows a backend pool behind the same tool signature. New file hearth/etc/backends.toml:

[[backend]]
name     = "omen-ollama"
endpoint = "http://127.0.0.1:11434"
api      = "ollama"
models   = ["qwen3-coder:30b"]
tags     = ["default", "code"]
revive   = "hearth start_ollama"            # existing summon tool

[[backend]]
name      = "am4-hermes"                    # the banked fire
endpoint  = "http://100.116.82.60:8090"
api       = "openai"
auth_env  = "HEARTH_HERMES_TOKEN"
models    = ["qwen3-30b"]                   # dual-B70 resident
tags      = ["big-context", "research", "second-opinion"]
occupancy = { conductor_worker = "am4-worker-1" }
revive    = "hearth wake_am4"

Routing policy, in order: (1) caller-pinned endpoint wins, unchanged; (2) tag match (task="research" → prefer B70s); (3) occupancy check — busy backends are skipped; (4) fall back to omen-ollama; (5) ledger the routing decision. The adapter difference (ollama /api/generate vs OpenAI /v1/chat/completions) is ~40 lines.

Lane 2 · Tasks (asynchronous, minutes) — research & simple builds

New HEARTH tool submit_task: writes a normal work item into the conductor's existing inbox (the same machinery that runs fleet builds), tagged origin=hearth and class=opportunistic. The conductor dispatches it to am4-worker-1 / omen-worker-1 exactly like any build; results land back as ledger events. Research briefs, doc summarization corpora, test-scaffold builds, retro drafts — all become fleet work items instead of frontier tokens. HEARTH adds a door; the conductor keeps the schedule.

Occupancy protocol (both lanes)

  • Probe: conductor API — any active dispatch on the backend's conductor_worker? Plus backend serve-truth (in-flight request count) where exposed. (v0 can ship with the conductor check alone.)
  • Lease: opportunistic work holds a 10-minute lease; renewal re-probes. Busy signal → no renewal → task re-queues. Inference calls are short enough to just finish.
  • Hysteresis: a non-resident model may load only after N idle minutes AND queue depth ≥ K (protects the DDR4 ceiling; vllama's OOM gates are the backstop).

Self-healing — the mechnet leveraging itself

Generalize what doorcheck proved today. fleet/inventory.toml checks grow an optional revive field naming a HEARTH tool or command. A mechnet-watchdog (scheduled task on OMEN, every 15 min: fleet_ping --json → for each down expected-up service, run its declared revive once → re-probe → record_event either way) turns the inventory into a self-healing declaration. Un-revivable failures surface on the ledger instead of being discovered mid-dispatch. OMEN is the right host: always-on, sees tailnet and VM network simultaneously.

What NOT to build

  • NO a second scheduler or queue store — the conductor exists.
  • NO tunnels or network plumbing — hermes is already reachable.
  • NO a model-lifecycle manager on AM4 — vllama is the mature layer; pull its truth, don't wrap it.
  • NO per-task custom configs — if a job needs config the backends.toml can't express, it's fleet work (lane 2), not a new lane.
  • NO autonomous frontier spend — banked fire burns sunk compute only; metered calls stay human-initiated.
