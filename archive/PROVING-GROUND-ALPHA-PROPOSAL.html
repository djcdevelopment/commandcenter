Part 1 · The pitch, human version

You already live the problem this thing measures: you run local models (Hermes on Apple Silicon) and you pay several online APIs for tokens, Claude included. Every day you make an implicit bet about which one to point at which job, and that bet is currently priced on vibes. The Proving Ground makes it priced on evidence: submit a task once, it fans out to every contender — my fleet (consumer GPUs running local models), your endpoints, the frontier APIs — and comes back with side-by-side outputs, quality scores from an automated judging loop, latency, token counts, and cost per contender. The whole run is traced end-to-end, so nothing is a claim; everything is a receipt.

This grew directly out of the Edgeless integration work — that's why you're getting the first invite. The same fleet that generates and scores images can run planning, refinement, document analysis, copy, and code tasks, and the interesting question stopped being "can it do the work" and became "how does local hardware stack up against metered APIs on a real workload, in public, with receipts." You get to be the first real workload that isn't mine.

A2 · Endpoint contract (what an L2 contender must provide)

RequirementLevelDetail
POST /v1/chat/completionsMUST OpenAI-compatible request/response. Non-streaming JSON is the baseline.

Streaming (SSE)MAY Optional. If declared, must be conformant (data: chunks, terminal [DONE]). If flaky, declare non-streaming — that's fine.
Declared metadataMUST Model ID(s), true context window, effective max concurrency, cost basis ($/1M tokens or sunk/electricity). Self-declared, labeled as such in reports.

  BASE_URL = {{https://your-endpoint.example}}
  MODEL    = {{model-id-to-validate}}
  API_KEY  = {{key-you-minted-for-the-harness}}

Judge each check PASS / FAIL / UNTESTED and keep the evidence (raw responses,

V1  DISCOVERY   GET $BASE_URL/v1/models with the key.

V2  COMPLETION  POST /v1/chat/completions, messages=[{user:"Reply with the
                single word: pong"}], max_tokens=8.
                PASS: 200, non-empty content, AND a usage block with plausible
                prompt_tokens/completion_tokens. usage missing = FAIL (hard).
V3  BOUNDS      max_tokens=16 on a prompt inviting a long answer; then a
                request with stop=["7"] asking it to count 1..20.
                PASS: completion_tokens ≤ ~20; generation stops at the stop
                sequence. Either ignored = FAIL.
V4  STREAMING   Same call with stream=true (skip → UNTESTED if not declared).
                PASS: valid SSE (data: chunks, terminal [DONE], parseable
                deltas). Malformed frames or missing [DONE] = FAIL.
V5  CONTEXT     Build a prompt ≈50% of the DECLARED context window: sentinel
    TRUTH       "ALPHA-7391" first line, filler text, sentinel "OMEGA-2486"
                last line. Ask: "state both sentinel codes".
                PASS: both echoed. First-only = silent tail truncation = FAIL
                (this invalidates every comparison — see F2).
V6  ERROR       Request model="does-not-exist-xyz".
    SHAPE       PASS: 4xx + JSON error object. 200, HTML, or a hang = FAIL.
V7  COLD/WARM   First request after ≥10 min idle vs an immediate second

V8  CONCURRENCY 3 simultaneous small completions.

                Serialized is acceptable IF declared as max_concurrency=1.

A4 · Failure catalog — what breaks, and why it matters

Failure pointCaught byWhy it matters
F1 · Missing usage blockV2 The harness can't count tokens → can't cost-account → the cost comparison (the entire punchline of the instrument) is broken for this contender. Hard blocker.
F2 · Silent context truncationV5 The model answers on a partial document without saying so. It scores unfairly low — or worse, wins on a task it never fully read. Either way the comparison is invalid, not just noisy. Hard blocker until declared window is corrected.
F3 · Fake-ready (accepts before loaded)V10, V7 The endpoint says 200-ready while the model is still loading; the first real request eats a 60–90s "latency" that is actually load time, or times out entirely. Skews every latency figure. Known failure class on local model servers — the harness treats "ready" as a claim to verify, but a contender that lies about readiness wastes run budget.
F4 · Nonconformant SSE streamingV4 Malformed frames or a missing [DONE] make the harness misread partial output or wait forever. Never fatal — declare non-streaming and it's a non-issue.
F5 · max_tokens/stop ignoredV3 Runaway generations inflate the contender's own latency and cost numbers and blow per-run budgets. The contender mostly hurts itself — but budget overruns stall the whole mission.
F6 · Undeclared concurrency = 1V8 The harness fans out in parallel; an endpoint that silently serializes (or errors on the 2nd connection) stalls the run. Declared max_concurrency=1 is perfectly fine — the harness serializes for you. Undeclared is the failure.
F7 · Errors as 200/HTML/hangsV6 Retry logic can't tell fatal from transient → wasted retries, false failure verdicts, poisoned scores. JSON errors with real status codes let the harness be fair to you.
F8 · Host sleep / power managementV7 (partly) macOS App Nap / sleep kills the endpoint mid-run and it reads as a contender failure. For run windows: caffeinate or equivalent. A dropped run is labeled infrastructure-failure, not model-failure — but it still burns the slot.
F9 · Unified-memory pressure / model swapV9 On Apple Silicon, another large model (or the principal's normal workload) sharing unified memory causes swap thrash: tok/s decays run-over-run. Not a blocker — V9 documents it so a slow-under-load result is explainable rather than mysterious.
F10 · Tunnel drops mid-stream— Funnel/cloudflared hiccups truncate a response; the harness retries within budget and labels the retry in the trace. Frequent drops → contender marked unstable. Nothing to validate up front; just know it's watched.
F11 · Hidden proxy to a metered APIhonesty An "endpoint" that quietly forwards to a paid cloud API with cost basis declared as sunk makes the economics readout lie. Reports label self-declared contenders for exactly this reason. Declare what's behind the URL; nobody audits your basement, but the label follows the claim.
